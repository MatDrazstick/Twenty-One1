BUG 2: Timer Force Draw Doesn't Continue
===========================================

PROBLEM:
--------
When the timer ran out and forced a player to draw, the game would not automatically 
proceed to the AI's turn. The game would wait for another user action before the AI 
could take its turn, causing the game flow to stall.

ROOT CAUSE:
-----------
The timer callback in Game.ts was calling playerDraws() but the async nature wasn't 
properly managed, and the game loop in interactiveGame.ts wasn't continuing after 
a forced draw.

FIX APPLIED:
------------
Location: /workspaces/Twenty-One/src/Game.ts (Lines ~494-510)

Improved the timer expiration logic to properly handle async forced draws:

```typescript
// Time's up - force action
if (this.turnTimeRemaining <= 0) {
  this.stopTurnTimer();
  
  const currentPlayer = this.players[this.currentPlayerIndex];
  const currentPlayerStayed = (this.currentPlayerIndex === 0) ? 
    this.player1Stayed : this.player2Stayed;
  
  // Bug 3 Fix: Check if player has already busted or stayed
  if (currentPlayer.isBusted || currentPlayerStayed) {
    console.log(`\n⏰ Time's up! Player has already ${currentPlayer.isBusted ? 'busted' : 'stayed'}.`);

    this.switchTurn();
  } else {
    // Bug 2 Fix: Force the player to draw and let it complete

    this.playerDraws().catch(err => console.error('Error forcing draw:', err));
```

The key improvement is that playerDraws() naturally switches turns when it completes,
allowing the game loop to continue with the AI's turn.

3. Rewrote updateKillMachine method (Lines ~418-450):
```typescript

  const actualMoveDistance = this.moveDistance + this.betModifier;
  
  // Number line: 0 = P1, 6 = center, 12 = P2
  if (roundWinner === this.players[0]) {
    // Player 1 wins, machine moves toward Player 2 (right on number line)

    // Check if player 2 has Bless and would die
    if (this.players[1].hasBless && newPosition >= 12) {
      console.log(`${this.players[1].name} uses Bless to avoid death!`);

      return;
    }
    
    this.machinePosition = newPosition;
    console.log(`Machine moves ${actualMoveDistance} toward ${this.players[1].name}`);
  } else {
    // Player 2 wins, machine moves toward Player 1 (left on number line)
    const newPosition = this.machinePosition - actualMoveDistance;
    
    // Check if player 1 has Bless and would die
    if (this.players[0].hasBless && newPosition <= 0) {
      console.log(`${this.players[0].name} uses Bless to avoid death!`);
      this.players[0].hasBless = false;
      return;
    }
    
    this.machinePosition = newPosition;
    console.log(`Machine moves ${actualMoveDistance} toward ${this.players[0].name}`);
  }
  
  // Calculate distances from machine to each player
  const distanceToP1 = this.machinePosition - 0;
  const distanceToP2 = 12 - this.machinePosition;
  console.log(`Machine position: ${this.machinePosition} (Distance to P1: ${distanceToP1}, Distance to P2: ${distanceToP2})`);
```

===========================================

PROBLEM:

ROOT CAUSE:
-----------
In the playerDraws() method, when a player busted, the code would switch turns but 
didn't provide clear messaging about what was happening. The round wouldn't properly 
continue if the other player still needed their turn.

FIX APPLIED:
------------
Location: /workspaces/Twenty-One/src/Game.ts (Lines ~218-241)

Improved the bust handling logic in playerDraws():

```typescript
    console.log(`You busted with a total of ${currentPlayer.calculateTotalScore()} (target: ${this.targetNumber})!`);
  }
  
  // Bug 5 Fix: Check if other player has also busted or stayed
  const otherPlayerIndex = 1 - this.currentPlayerIndex;
  const otherPlayer = this.players[otherPlayerIndex];
  const otherPlayerStayed = (otherPlayerIndex === 0) ? this.player1Stayed : this.player2Stayed;
```

Key improvements:
- Explicitly check if the other player is done before ending the round
- Don't mark busted player as "stayed" - they're out due to bust
- Provide clear message that the player is out and whose turn it is next
- Properly switch turns so the other player can continue

CRITICAL ISSUE DISCOVERED:
After the previous fix, a more serious issue was found: the user could type
commands during the AI's turn and act FOR the AI. The AI was not acting

ANALYSIS:
The game loop structure had a fatal flaw:
```

  if (both players done) continue;
  if (AI turn) { executeAITurn(); continue; }
  // Human turn code here...
}
```

When executeAITurn() returned early (because AI already stayed/busted), it
would return to the game loop, but NOT continue the loop. This caused execution
to fall through to the human turn code, showing prompts and accepting user
input even though it was the AI's turn!

COMPLETE FIX:
Added a guard that checks if the current player (AI or human) has already
completed their turn BEFORE processing the turn. If they have, manually switch
to the other player and continue the loop.

Code Location: /workspaces/Twenty-One/src/interactiveGame.ts

```typescript
// Bug 2 Fix: Check if current player is already done before processing turn
const currentPlayerDone = (game.currentPlayerIndex === 0) ? p1Done : p2Done;
if (currentPlayerDone) {
  // Current player is done, switch to other player
  console.log(`${currentPlayer.name} has already completed their turn.`);
  game.currentPlayerIndex = 1 - game.currentPlayerIndex;
  continue;
}

// If it's the AI's turn, execute AI turn
if (game.currentPlayerIndex === 1) {
  await game.executeAITurn();
  continue;
}

// Human player's turn (only reached if it's human's turn AND not done)
```

This ensures:
1. Done players are skipped and turn switches automatically
2. AI turn code is only reached if AI hasn't acted yet
3. Human turn code is only reached if it's human's turn AND they haven't acted
4. No fall-through from AI turn to human turn prompts

===========================================
BUG 7: Wrong Winner When Both Players Bust
Date: February 10, 2026
===========================================

PROBLEM:
--------
When both players bust (go over 21), the game incorrectly determined the winner.
The player who was FURTHER from 21 was being declared the winner instead of the
player who was CLOSER to 21.

USER REPORT:
"the round ended and i had a score of 23 while the AI had a score of 28. 28 is 
further away from 21, so I should have won - but the AI was flagged as the winner 
instead"

Example:
- Player 1: score 23 (over by 2)
- Player 2: score 28 (over by 7)
- Expected: Player 1 wins (closer to 21)
- Actual: Player 2 wins ❌

ROOT CAUSE:
-----------
Location: /workspaces/Twenty-One/src/Game.ts (endRound method, Line 449)

When both players bust, the winner determination logic was:
```typescript
if (score1 > this.targetNumber && score2 > this.targetNumber) {
  // Both bust - closer to target wins
  roundWinner = (this.targetNumber - score1 < this.targetNumber - score2) ? player1 : player2;
}
```

The problem: Using subtraction without absolute value on negative differences!

With the example scores:
- targetNumber = 21, score1 = 23, score2 = 28
- Calculation: (21 - 23) < (21 - 28)
- This becomes: (-2) < (-7)
- Result: FALSE (because -2 is GREATER than -7)
- So player2 wins incorrectly!

The issue is that -2 < -7 is false in mathematics, but we want to compare the
distances (absolute values): |−2| < |−7| which is 2 < 7, which is true.

FIX APPLIED:
------------
Location: /workspaces/Twenty-One/src/Game.ts (Line 449)

Changed the comparison to use Math.abs() to get absolute distances:

```typescript
if (score1 > this.targetNumber && score2 > this.targetNumber) {
  // Both bust - closer to target wins (Bug 7 Fix: use absolute value)
  roundWinner = (Math.abs(this.targetNumber - score1) < Math.abs(this.targetNumber - score2)) ? player1 : player2;
}
```

Now with the example:
- Calculation: |21 - 23| < |21 - 28|
- This becomes: 2 < 7
- Result: TRUE ✓
- Player1 wins correctly!
